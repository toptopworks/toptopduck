# result_N lifecycle: soft-invalidate, windowed LLM view, GC cap

## Decision

`result_N` 生命周期三档：

1. **删除 = 级联失效（soft delete），非级联硬删**。删 `result_N` → 它及所有传递下游标记 **stale**：保留可见（历史可查）、**不可再被新 SQL 引用**、不进 LLM 可见工作集。用户可基于未失效的上游重算。
2. **LLM 可见工作集 = 近窗裁剪**。每轮 prompt 含：全部源 Dataset 全 schema + **最近 K=20 个活跃 `result_N` 的全 schema（含样本）**；窗口外的 `result_N` 只发**表名 + 列名 + 一句话摘要**（=该步提问摘要；不发样本、不发冗长类型），使 ADR-0010 的自然语言重定向对老结果仍可工作——仅机器名+列名不足以让 LLM 把"去重后的数据"这类自然语言指代映射到 `result_N`，须附摘要。
3. **硬上限 = 只 GC stale**。会话内 `result_N` 总数超 **M=100** 时，自动 GC 最老的 stale；**活跃的永不自动删**（仅用户显式删才动）。

## Context

ADR-0010 让 `result_N` 可分支，ADR-0003/0009 早标注却未定的两件事就此绑定：删除一个被依赖的 `result_N` 的语义，以及工作集 schema 随步数无限膨胀的 token/质量代价。

## Why

1. **soft 失效比硬级联删安全**——不无声丢弃用户可见的中间成果（保住 ADR-0010 的可检视历史信任）；比"拒绝删除"灵活。
2. **近窗裁剪把无界的 schema 喂给 LLM 收成有界**——控 token 成本、减少无关 schema 对 SQL 生成的干扰；窗口外只留表名+列名，保住重定向能力，不牺牲表达。
3. **GC 只动 stale**——活跃成果永不被自动回收，用户对"我算出来的东西不会消失"有确定预期。

## Considered options

- **删除即硬级联删**：丢用户可见成果，破坏历史信任。**否决**。
- **拒绝删除有依赖的 result**：僵，用户被卡住。**否决**。
- **LLM 视图按引用热度/LRU 裁剪**：更聪明但复杂，v1 过度。**否决（v1）**。
- **pin 钉住常驻全 schema**：好增强，v1 不做（YAGNI）。

## Consequences

- stale 是用户可见状态，须在历史视图显式标注"此步因上游删除已失效"。
- 窗口外 `result_N` 的重定向依赖"表名+列名+一句话摘要"仍在 prompt——摘要（提问摘要）让 LLM 把自然语言指代映射到 `result_N`，列名是写对 `FROM` 的最小信息。
- K=20 / M=100 为默认，可调；分支激增场景下 M 是安全阀。
- 用户对"活跃成果不会被自动删"是明确契约。
- **被 ADR-0023 校准**：窗口计数口径从"按 `result_N`（K=20）"统一为"按轮次（N=20）"——见 ADR-0023；语义不变，M=100 不受影响。
