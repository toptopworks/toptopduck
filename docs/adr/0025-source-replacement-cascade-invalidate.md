# Source replacement: re-upload replaces the source, cascades soft-invalidate its derivatives

## Decision

「重新上传」一个源 Dataset = **替换该源**（非新增）：新快照 copy-in **接管旧源表名**，旧快照丢弃。旧快照的全部派生 `result_N` **级联走 ADR-0013 的 soft 失效（stale）**——保留可见、不可再被新 SQL 引用、不进 LLM 工作集；用户可基于新源重算。新派生取 `result_{max+1}`（编号不重用，ADR-0022）。

「替换此源」与「上传新源」由**入口区分**：在某源上点"重新上传"= 替换语义；走文件选择器上传 = 新源（撞名则去冲突，ADR-0022）。不让同名上传自动猜。

## Context

ADR-0012「用更新的源重算 = 显式重新上传（新会话内快照）」、ADR-0020「类型错 = 重新上传」都假设重新上传存在，却未定：旧快照替换还是并存？旧派生 `result_N` 作废还是失效？同名源命名空间怎么办？源变了，其派生血缘必然受影响，行为必须钉死。

## Why

1. **替换而非并存 = 贴合用户意图**：重新上传的语义是"我要用新版本这个源"，不是"加一个并列源"；并存会留无用旧快照占内存/混淆。想并列就用多源上传（ADR-0022）。
2. **派生级联 soft 失效复用 0013 既有机制**：源变了，基于旧快照的派生失真，必须断；soft 失效（保留可见、不可引用）是源变更的最小代价——不丢用户可见成果（0013 已否决硬级联删），也不硬留失效派生误导。
3. **入口区分消除同名歧义**：同名文件上传时"替换 vs 新增"是真实歧义，让系统猜会误操作（覆盖了用户想保留的源）；显式入口把语义交给用户。
4. **编号不重用**（0022）：失效派生保留原名，新派生取新号，命名空间不撞。

## Considered options

- **重新上传 = 新增并存源**：留无用旧快照、混淆命名空间。**否决**。
- **派生硬级联删**：丢用户可见成果，破坏 0013 历史信任。**否决**。
- **派生保留不失效**：源变了派生失真却仍可引用 → 误导。**否决**。
- **同名上传自动猜替换/新增**：误操作风险（覆盖想保留的源）。**否决**。

## Consequences

- **延伸 ADR-0013**：soft 失效机制不仅服务于"用户删除 `result_N`"，也服务于"源被替换导致的派生失效"——同一机制，两种触发。
- 重新上传是源变更的唯一路径（呼应 0012 copy-in 冻结、0020 类型错重传）；会话内无路径改源。
- 失效派生在历史视图须标注"因源已更新而失效"（0013 stale 标注的扩展措辞）。
- 入口区分"替换/新增"是上传流 UX 必要交互（与 ADR-0015 引导加载、0020 类型暴露同属上传流）。
