# Application session model: multiple isolated in-memory sessions (per-session DuckDB), soft cap

## Decision

v1 应用支持**多个并行会话**——每会话一个**独立内存 DuckDB 实例**（ADR-0012）、各自**单飞行查询**（ADR-0021）、各自 `memory_limit`（ADR-0005）。会话是**物理隔离的封闭工作区**：一个会话的 `result_N` 在另一会话不可见、不可引用；跨会话搬数据走**导出（ADR-0004）→ 重新上传**。设**全局会话数软上限提示**（超阈值提醒内存压力），但**不硬限**。UI 形态（单窗口多 tab vs 多窗口）为 UX 子决策。

## Context

ADR-0012（纯内存会话）、ADR-0010（线性 thread）、ADR-0021（会话内单飞行）都讲"会话"单数行为，但应用层一次能开几个会话——单会话还是多会话——从未定义。而 ADR-0008 的 Tauri 天然支持多窗口/webview。这决定「换数据集/开新分析是否丢当前工作集」「多会话如何分内存」。

## Why

1. **多会话消除"新分析=重置=丢工作集"硬伤**：单会话强制用户每次换数据集/开新分析就重置，丢失链式迭代的中间成果——对非技术用户（ADR-0001）的 notebook 心智是硬伤。多会话用一点内存换掉它。
2. **物理隔离是独立内存实例的自然结果**：每会话独立 DuckDB 实例 → 表/`result_N` 天然互不可见，无需额外隔离机制（KISS）。
3. **单飞行/内存上限各自独立**：ADR-0021 单飞行、ADR-0005 `memory_limit` 都是"会话内"语义，多会话下各自适用，无需重新设计。
4. **软上限不硬限**：硬限会把"对比 3 个数据集"卡死；软提示把内存决策交给用户，符合"用户主动开=主动承担"。

## Considered options

- **v1 单会话**：最省内存、最简单，但强制"新分析=重置=丢工作集"，对链式迭代体验是硬伤。**否决**。
- **多会话 + 跨会话引用**：更灵活（会话间共享 `result_N`），但破坏物理隔离、引入跨实例一致性/命名空间问题，v1 过度。**否决（v1）**。
- **多会话 + 硬上限**：控内存，但卡死"对比多数据集"真实需求。**否决**。
- **会话持久化（多会话 + 可恢复）**：违背 ADR-0012 纯内存/不做持久化；v1 不做。**否决（v1）**。

## Consequences

- 应用层须管理多个独立内存 DuckDB 实例的生命周期（创建/销毁随会话开关）；每实例各自单飞行（0021）、各自 `memory_limit`（0005）。
- 跨会话搬数据 = 导出（0004）→ 重新上传（0012/0025）；无共享 `result_N` 通道。
- 软上限提示为 UX 必要交互；硬上限明确不做。
- UI 形态（tab vs 多窗口）为 UX 子决策，不阻塞架构；倾向单窗口多 tab（对非技术用户友好）。
- 多实例内存翻倍是已知代价，由各自 `memory_limit`（0005）+ 软上限提示兜底。
- **被 ADR-0034 取代（持久化部分）**：本 ADR Considered options 中「会话持久化（多会话+可恢复）…v1 不做。否决（v1）」被反转——会话现为持久化单位（recipe-only、可移植 .duck、eager 重放、每轮原子写，见 0034）。保留：多会话**运行态**内存隔离、各自单飞行（0021）、各自 `memory_limit`（0005）——这些是打开态语义，持久化不改变它们；本 ADR 不再以"纯内存 / 关即重置"框定会话。
