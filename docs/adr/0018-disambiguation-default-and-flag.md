# Disambiguation: default to active dataset + flag the assumption; clarify only when the active default doesn't hold

## Decision

列名 / 连接键歧义时：
- **默认用当前表（ADR-0010）+ 最合理解读执行，并显式标注假设**（如"按 **orders.name（产品名）** 汇总；若指 customer 名请说"），让用户一眼纠偏。
- 仅当**当前表默认都站不住**（列不在当前表、或 ask 明确跨表且键/字段多重可能、无法合理推断）时，才**升级为澄清提问**——返回一条澄清问题而非 SQL（**窄门槛**）。
- 连接键歧义同策：默认选最像主/外键的列 + 标注，高不确定则澄清。
- 契约支撑：ADR-0009 扩展为可返回"澄清/拒绝"**文本态**（无 SQL）。

**核心哲学**：静默的错误 > 暴露的假设 > 打断式追问——优先"做了 + 标注"，次选"问"，**绝不**硬猜。

## Context

工作集常有多个 Dataset 含同名列。ADR-0010 的当前表已消解大部分歧义（当前表内操作无关他表）；残留尖刺是**跨表/连接**、及**当前表默认也不确定**时的行为。LLM 若硬猜 → 自信的错误（信任杀手，见 ADR-0017）。

## Why

1. **静默猜 = 自信错误**，与 0017 同一信任命门；标注假设让纠偏成本极低。
2. **窄门槛澄清**把打断摩擦留给真正无法合理推断的情形，避免"动不动就问"的烦人 UX。
3. **契约加文本态**统一支撑 0017 拒绝 + 0018 澄清——同一机制服务两种"不产 SQL"的轮次。

## Considered options

- **宽门槛：一丝歧义就问**：最安全但打断多、烦。**否决**。
- **永远硬猜不标注**：自信错误风险，信任杀手。**否决**。
- **永远问、不默认推进**：UX 烦、链式体验断裂。**否决**。

## Consequences

- ADR-0009 契约扩展为 `{ sql, viz?, assumption? }` | 文本态（澄清 / 拒绝）；假设标注走 `assumption` 字段。
- 结果展示须能挂"假设标注"（与 ADR-0010 历史视图的纠偏抓手一致）。
- 澄清门槛由 system prompt + 谓词引导（软保证，同 0017）。
