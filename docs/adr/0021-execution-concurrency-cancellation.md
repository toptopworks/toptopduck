# Execution concurrency & cancellation: single in-flight query, user-cancellable, no concurrent queries

## Decision

v1 会话内**一时刻最多一条查询在执行**（线性延伸 ADR-0009 的"一轮一条 SQL"）。查询进行时**禁用输入框**（或入队待当前完成再提交）。用户可**随时取消**当前查询（前端"停止"按钮 → DuckDB query interrupt）；取消或语句超时（ADR-0005）后**当前轮作废**——不产 `result_N`、工作集不变、不计入 ADR-0013 步序。**不支持并发查询**。

## Context

ADR-0009 定"一轮一条 SQL"，但未定执行层：长查询（大表 + 复杂 SQL）可能跑很久甚至触 0005 语句超时，用户能否中途取消？查询进行时新提问如何处理？无 ADR 定义。卡死对非技术用户（ADR-0001）是灾难；并发则引入 result_N 命名竞争、工作集一致性、内存争抢（0005 memory_limit 单实例共享），且违背 0010 线性 thread 心智。

## Why

1. **单查询取消防卡死，成本低**：DuckDB Rust binding 支持 query interrupt；前端"停止"按钮即可；作废语义干净（不产 result_N、工作集不变）。
2. **不并发契合线性心智**：会话是单用户线性 thread（0010）；"一轮一条 SQL"自然延伸为"一时刻一条查询"。禁用输入/入队避免命名竞争与工作集一致性问题（YAGNI）。
3. **超时与取消统一路径**：0005 语句超时与用户取消都走"当前轮作废"，执行层只有一种失败/中断语义，护栏最简。
4. **并发是未来扩展点**：若未来要支持（如后台预计算），需另开 ADR 处理命名/一致性/内存隔离。

## Considered options

- **v1 不做取消，纯靠 0005 超时兜底**：最简，但用户被卡几十秒无法干预，非技术用户灾难。**否决**。
- **支持并发查询**：最灵活，但 result_N 命名竞争、工作集一致性、内存争抢，且违背 0010 线性心智。**否决（v1）**。
- **取消后保留部分结果**：复杂且语义混乱（部分 result_N 进工作集？），违背"一轮一 result_N"。**否决**。

## Consequences

- **校准 ADR-0005**：语句超时与用户取消（ADR-0021）走同一"当前轮作废"路径（不产 result_N、工作集不变）。
- **校准 ADR-0009**："一轮一条 SQL"延伸为"一时刻一条查询"；查询进行时禁用输入/入队。
- 执行层须实现 query interrupt + 前端"停止"按钮；作废轮不产 result_N、不计步序，可在历史视图标记"已取消"（透明）。
- 并发查询为已知 v2 扩展点，需先解决命名/一致性/内存隔离。
- **被 ADR-0028 校准**：「不计步序」精确化为「步序 = `result_N` 编号（单一轴，ADR-0022）；cancel 轮恒可见、占 N=20 槽（ADR-0023）、但不动编号」；轮次仅有时序位置、无独立步号计数器、不向用户展示步号。
