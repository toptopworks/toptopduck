# Privacy boundary refinement: real samples by default, user-controlled, honestly disclosed

## Decision

精化 ADR-0006 的外发载荷：**默认发送真实样本行**（非零样本）。具体——每个 Dataset 默认发 **3 行**真实样本；用户可**按数据集**开关 / 减少 N，可**按列**把某列标记为"只发类型、不发样本"；**列名默认随 schema 发送**，但敏感列名同样可标记为"只发类型、不发名"。面向用户的隐私边界**诚实重述**为："完整数据集不出本机；默认向 LLM 提供商发送 3 行样本与列名——你可关闭、减少、或按列脱敏。" 该边界的可信度**部分依赖 provider 政策**（BYOK API 调用默认不进入训练），UI 须披露此依赖。

## Context

ADR-0006 把"样本量可调/可关"留作后果、未给默认。尖锐问题：样本行就是**私密数据集的真实数据**，故"数据不出本机"技术上成立但对用户是**话术漏洞**；列名本身也泄露 PII；边界还部分攥在 provider 留存/训练政策手里。需给一个**具体默认 + 诚实框架**。

## Why

1. **schema-only 在真实脏数据上可测地劣化 SQL 质量**（日期格式、null 模式、单位、脏值）——而质量是 ADR-0006 自陈的"产品命脉"。故零样本方案 (a) 否决。
2. **真样本 + 用户管控**：既保质量、又把选择权交还用户；诚实披露把"隐藏的洞"变成"明示可调的边界"——是信任卖点，而非负债。
3. **自动 PII 探测 (c) 对 v1 是 YAGNI**：按列**人工标记**即可覆盖，无需自建 PII 识别器。
4. **列名 PII 必须进同一控制面**，否则边界照漏。

## Considered options

- **(a) 零样本，只发 schema + 类型 + 合成示例值**：隐私最强，但质量劣化，违背"命脉"定位。**否决**。
- **(b) 真样本 + 强管控 + 诚实披露**：质量与隐私兼得，用户掌控。**采纳**。
- **(c) 自动 PII 探测 + 脱敏采样**：体验最佳但 v1 偏重。**否决（v1），留作未来增强**。

## Consequences

- 默认发 3 行真实样本——隐私 claim **必须重新措辞**，UI 明示"默认发送样本与列名、可控"。
- 敏感列 / 敏感列名的"只发类型"控制是**必需功能**（非可选）。
- 边界可信度部分依赖 provider 政策——UI 须披露，**不能假装纯本地保证**。
- 自动 PII 脱敏 (c) 为已知增强点，不阻塞 v1。
