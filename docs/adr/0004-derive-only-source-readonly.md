# Capability boundary: derive-only, source data is read-only

## Decision

会话对**源 Dataset 严格只读**。一切"处理"**派生**为新的 `result_N`（物化进工作集），原件永不修改。**导出 (Export)** 为显式用户动作——把某个 `result_N` 另存为 CSV/Parquet/Excel 到磁盘。禁止对源 Dataset 的 DML/DDL。

## Context

"数据处理"含义模糊：**派生**（产出新表）vs **变异**（覆盖原件）。需框定能力边界，并决定安全护栏形态。

## Why

1. **保护原始文件 = 桌面数据工具的信任命门**。AI 改坏用户真实数据 → 信任归零。派生模型从架构上保证原件永不被碰。
2. "处理"被完整满足：清洗/变换/计算均产出新表，原件无损；用户得到结果而非破坏。
3. 爆炸半径最小：只读 + 显式导出，护栏简且稳。
4. 可追溯：每步派生是可检视的 `result_N`，数据血缘完整。

## Considered options

- **全变异**（允许 UPDATE/DELETE、覆盖原文件）：像 ETL 一步到位，但可能损坏用户原始数据——桌面工具不可接受。**否决**。
- **纯只读、无导出**：最安全，但连结果都存不下来，能力 crippled。**否决**。

## Consequences

- 用户想"原地改文件"需多一步：导出后自行覆盖。
- 护栏需在 SQL/引擎层强制只读与禁用危险函数（见 ADR-0005）。
- 导出格式与目标路径策略需单独定义。
