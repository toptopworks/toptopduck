# Capability boundary: SQL-only v1 scope; out-of-scope requests are refused with an in-scope alternative, never faked

## Decision

**v1 能力边界**（SQL / DuckDB 原生范围内）：
- **IN**：关系查询、聚合、清洗（去重/类型转换/字符串/正则/null）、连接/合并、Pivot、格式转换、描述性统计（`corr`/`covar_*`、简单线性回归 `regr_*`、`median`/`quantile_*`、`stddev`/`variance`、`skewness`/`kurtosis`、`mad`）、异常值检测（z-score/分位）、排名/窗口/Top-N。
- **OUT**：预测/时序建模、ML（聚类/分类/推荐）、语义文本分类、假设检验（p/t-test）、优化求解、任意自定义变换。

**越界行为 = 拒绝 + 诚实解释 + 提供 in-scope 替代**，绝不用天真方法（如线性外推冒充"预测"）糊弄。能力边界写进 **system prompt**，模型主动拒绝越界而非编造 SQL。DuckDB 原生方法（相关/回归/异常值）属 IN，正常使用并**如实标注方法名**。越界检测是**软保证**（能力边界非安全/隐私边界——后者仍引擎级硬保证，见 ADR-0005）。

## Context

CONTEXT 原"解决日常数据分析问题"对非技术用户**过度承诺**（他们心中的"日常"含预测/ML）。更尖锐的是：LLM 为"满足"越界请求会用天真方法产出**自信的错误答案**（线性外推当预测）——对数据工具是信任灾难。且 DuckDB 实际原生统计能力比 ADR-0002"不做统计建模"的措辞更广。需画准边界 + 定义越界行为。

## Why

1. **自信的错误答案 ≫ 诚实的"做不到"**——对数据工具，错误比无能力更伤信任；拒绝 + in-scope 替代是更负责任的 UX。
2. **DuckDB 原生统计（`corr`/`regr`/`quantile`）是真能力，应纳入 IN**——边界比 0002 更准，不白白放弃。
3. **语义文本操作需 data→LLM**，超 ADR-0011 隐私边界，v1 明确 OUT。
4. **能力边界软保证可接受**：越界最坏情况是用户换说法绕过得到一条朴素 SQL——是 UX/honesty 问题，非安全/隐私，无需引擎级硬保证。

## Considered options

- **越界用天真方法 + 标注"非正式"**：更有用但有误用风险、且模糊边界。**否决**。
- **不定义边界、靠 LLM 自由发挥**：自信错误答案风险，承诺失信。**否决**。
- **把 DuckDB 原生统计也划 OUT（与 0002 措辞一致）**：白白放弃真能力。**否决**。

## Consequences

- CONTEXT 产品描述重写，去掉过度承诺。
- system prompt 须含能力边界 + "拒绝 + 替代"行为模板；越界拒绝须配 in-scope 替代建议（graceful，非纯拒绝）。
- 语义文本分析为已知未来能力，需先解决隐私路径（与 ADR-0011 联动）。
- 真正的 ML/预测/逃生舱仍待 ADR-0002 的未来 ADR。
- **prompt 注入是同源软保证的已知局限**：ADR-0011 发真样本使样本/列名成为 LLM 输入，可被注入绕过本 ADR 的能力边界（及 0010/0018 的解析）。v1 姿态 = 与越界同级软保证，最小防御三件套（system prompt 声明样本为不可信数据、prompt 数据/指令物理分隔、复用 ADR-0005 行数上限节流）；不做自动检测/消毒（YAGNI，且消毒破坏 0012 源不可变）。最坏情况 = 错误/越界结果（可感知纠偏），非数据破坏（0005 兜底）。
