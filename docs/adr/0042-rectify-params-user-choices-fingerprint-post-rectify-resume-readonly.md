# Rectify params in recipe: user choices only (not result snapshot); content fingerprint = post-rectify snapshot hash; resume read-only, fix via re-upload

## Decision

1. **规整参数 = 用户的显式规整决策**（表头行号、跳过行范围、sheet 选择、列类型覆盖——ADR-0015 引导加载 / ADR-0020 类型暴露时用户拍板者）。**自动规整算法**（表头探测、解合并单元格、公式取缓存值）**不进参数**——它是代码，resume 重跑当前版本。

2. **内容指纹 = 规整后 copy-in 快照的 hash**（精确化 ADR-0035「源 hash」）——校验落在 recipe 要保证的最终产物（规整后那张表），而非源文件原始 bytes。

3. **resume 重建 + 校验**：读源文件 → 应用规整参数 + 当前算法 → 产出快照 → 校验快照 hash == recipe 记录。符则静默进入；不符走 ADR-0035 漂移降级。**算法跨版本变化**（v1.1 改了表头探测启发式）→ 同源同参数产出不同快照 → hash 不符 → 漂移检测**自然命中**：源漂移与算法漂移走同一条 0035 降级路径，无新机制。

4. **换源 = 对新源重新规整**：走 ADR-0015/0020 流程，产新规整参数 + 新快照 hash；旧规整参数随旧快照丢弃。规整参数是源引用的一部分（ADR-0034），换源 = 替换整个源引用（路径 + 规整参数 + 指纹）。

5. **resume 后规整参数可见只读**；修正规整的唯一路径 = **重新上传 = 换源**（ADR-0025）。换源判定依据精确化 = **快照变了**（规整后 hash 不同），非「文件变了」——**同源文件 + 不同规整也算换源**（新规整参数 + 新指纹接管旧引用名 + 级联失效派生）。

6. **CSV/JSON v1 无用户规整参数**（ADR-0020：纯 DuckDB 自动类型推断、不做引导式类型确认）；其规整参数基本空，copy-in 快照的确定性纯靠规整后 hash 校验。

## Context

ADR-0034 的 recipe 结构「源引用 = 路径 + 规整参数 + 内容指纹」里，**「规整参数」是黑盒**（0015 的交互产物如何序列化未定）、**「内容指纹」语义未钉**（源文件原始 hash 还是规整后快照 hash）。规整是「源文件 → copy-in 快照」的转换，而 0034 承诺 recipe 是工作集的「完整无损」描述、resume 重建出**完全相同**的快照（ADR-0012）——这个转换的**确定性保证从哪来**未定。是 0015 自陈「导出路径策略另定」已被 ADR-0038 收口、但其规整交互产物如何进 recipe 仍未闭合的依赖。

## Why

1. **KISS + 兑现 0034 完整无损**：规整参数轻量（只记用户决策），配规整后快照 hash 校验，确定性保证由「重放 + 校验」兜底，而非靠记全量规整结果。
2. **漂移统一捕获**：源文件改 / 算法版本变，都表现为「重放快照 hash ≠ 记录 hash」，同一 0035 降级路径，无新机制、无新代码路径。
3. **可见只读避免级联陷阱**：会话内改规整会触发 ADR-0025 级联失效，用户难理解「为何改个加载设置炸了整条分析链」；修正统一走重新上传（0012/0020 既有路径），KISS/DRY。
4. **边界假设显式化**：本模型依赖「自动规整算法是确定性的（给定输入 + 版本 → 确定输出）」。若将来规整引入不确定性（如 ML 启发式），须把结果也记进参数——届时另开 ADR 校准本决策。

## Considered options

- **规整参数记完整结果快照（版本无关）**：自包含但 recipe 膨胀；且快照 hash 校验已能捕获算法漂移、诚实降级，无需全量记结果。**否决**。
- **内容指纹 = 源文件原始 bytes hash**：检测不到「算法版本变致规整结果变」；且 recipe 要保证的是规整后快照，校验应落在产物上。**否决**。
- **resume 可见可编辑（会话内改规整）**：触发 0025 级联失效的隐式陷阱；且是「既触发级联又不算换源」的半新操作，语义模糊。**否决**。
- **换源继承旧规整参数**：新源结构不同（表头行变了），旧参数大概率无效。**否决**。
- **算法升级时静默用新算法重算老文件**：静默产出微妙不同的快照，违 ADR-0011/0017/0030 诚实脊柱。**否决**。

## Consequences

- **精确化 ADR-0034**：recipe 源引用的「规整参数」「内容指纹」语义钉死——规整参数 = 用户规整决策（非结果快照）、内容指纹 = 规整后快照 hash（非源文件 hash）。0034 Consequences 已追加指针。
- **精确化 ADR-0035**：内容指纹 = 规整后 copy-in 快照的 hash；算法漂移与源漂移同走本 ADR 漂移降级。0035 Consequences 已追加指针。
- **精确化 ADR-0025**：换源判定依据 = 快照变（含同源文件不同规整），非文件变。0025 Consequences 已追加指针。
- **延伸 ADR-0015**：引导加载（表头行/跳过行）的用户交互产物 = 进 recipe 的规整参数；resume 后可见只读，修正 = 重新上传（= 换源）。0015 Consequences 已追加指针。
- **延伸 ADR-0020**：CSV/JSON v1 无用户规整参数（纯自动类型推断）；其快照确定性靠规整后 hash 校验。0020 Consequences 已追加指针。
- **延伸 ADR-0012**：copy-in 快照的 resume 重建路径钉死 = 读源 → 应用规整参数 + 当前算法 → 产出快照 → 校验规整后 hash（不符走 0035 降级）。0012 Consequences 已追加指针。
- **不改 CONTEXT**：规整参数是 recipe 的结构字段（实现细节），非用户面领域术语；用户面概念「规整」已在 CONTEXT「数据集」条目 + ADR-0015 覆盖。CONTEXT 须 devoid of implementation details。
- 实现侧：规整参数序列化（格式相关——Excel 有、CSV/JSON 空）、规整后快照 hash（save/resume 各一次流式）、resume 重放 + 校验、规整参数只读视图、换源对新源重新规整。
- **边界假设**：自动规整算法须确定性；若 v2 引入不确定性（ML 启发式），须把结果记进参数，另开 ADR 校准本决策。
