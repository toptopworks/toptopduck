# Stale derivatives after source replacement: not recomputed in v1 (dead turns) — user re-asks; explicit recompute deferred to v2

## Decision

1. **v1 换源级联失效的派生 = 终局死轮**：不重算、不复活。用户换源（ADR-0025）后想得到等价结果，走**重新提问**——LLM 基于新源重新生成 SQL、产新 `result_N`（编号不重用，ADR-0022）。recipe 保持"当前态、非历史账本"（ADR-0034）。

2. **失效 SQL 不被系统自动复用**：失效轮的 SQL 留在 recipe 的**对话历史展示部分**（全量、纯追加，ADR-0034），按既有窗口规则服务——**窗口内**（近 N=20 轮）发全量 SQL 供 LLM 参考（ADR-0023）、**窗口外**不发（ADR-0023）。但系统**不**自动取它重跑、不**复活**旧 `result_N`。

3. **理由精确化（换源 ≠ 删源）**：ADR-0025 定换源 = 新快照**接管旧源引用名**（ADR-0037：引用名是稳定身份、换源只换 backing）。故旧 SQL 的 `FROM` 旧源名，换源后**物理上指向新数据**——失效 SQL 其实"能跑"。这与**删源**根本不同（删源引用名消失、数据真不可得，才是真死轮）。**v1 不重算换源失效派生，不是因为不可能，而是不做"失效子链重放"子系统**（见 Why 2）。

4. **显式重算机制留 v2**：依赖链回溯 + 按序重跑 + 断点处理（本质是失效子链的 resume 重放，ADR-0034）是 v2 扩展点，与并发查询（ADR-0021）、加源并发（ADR-0040）同列"v1 不做、留待真需求"。

## Context

ADR-0040 在 Consequences 标了未闭合依赖："换源后失效轮 SQL 被裁出可重放链（0034）但留在对话历史；用户说'用新源重算刚才那个'——LLM 怎么取到那条失效 SQL？" 钻进去发现它取决于一个更上游的能力边界：失效派生能不能"复活"。又发现 ADR-0034"换源后旧源可能不可得 → 失效轮无法重算 → 从不重算死轮"的措辞把换源 / 删源混为一谈——换源因引用名被接管而 SQL 物理可跑，理由需精确化。是源变更维度（ADR-0040 延伸）的收口决策。

## Why

1. **保 recipe 纯粹**：复活旧 `result_N`（编号恒定、backing 换成新数据）让"backing 可变"——与 ADR-0022"编号永不复用"、ADR-0037"引用名恒定 / backing 不可变"的语义咬合处须重新立契约。复杂度不成比例于收益。
2. **依赖链重放是真子系统**：失效派生常依赖其他 stale 表（`result_5` 依赖 `result_2`，后者也失效）。"重算"须回溯依赖子链、按序重跑、任一断则停——本质是"失效子链的 resume 重放"（ADR-0034），是一整套子系统，v1 不做（YAGNI）。
3. **用户重新提问可接受**：换源语义是"用新版本这个源"，用户脑中 / 对话历史里仍有原逻辑；重新提问时 LLM 还能在窗口内看到失效 SQL 作参考（既有规则，无需新机制）。非技术用户（ADR-0001）重新描述一句的负担，低于为"一键重算"养一套子系统。
4. **诚实说不（呼应 ADR-0017）**：与越界拒绝同属"明确边界"——但此处是"v1 不做"非"永不做"，留 v2；不静默假装支持。

## Considered options

- **路 Y：复活失效派生（重跑旧 SQL、backing 换新源）**：保 `result_N` 编号，但 backing 可变语义与 0022/0037 咬合要重立契约；且依赖链 stale 仍须重放子系统。**否决（v1）**。
- **显式"取旧 SQL 重跑"开新轮（不复活、复用 SQL 文本）**：避开了 backing 可变，但旧 SQL 若引用其他 stale 表（`result_2`）仍跑不通 → 一样要依赖链重放。子系统省不掉。**否决（v1）**。
- **v1 就做依赖链重放**：真需求未证（换源后多少用户要精确复刻 vs 重新提问）；子系统成本高。**否决（v1，YAGNI）**。
- **静默自动重算失效派生（换源即重跑）**：违背 0034"按当前态组织"、破坏 stale 可见性（用户不知哪些被自动换了 backing）；不诚实。**否决**。

## Consequences

- **精确化 ADR-0034**：本 ADR「换源后旧源可能不可得 → 失效轮无法重算 → 从不重算死轮」的理由分场景——**换源**接管引用名、新源可得、旧 SQL 物理指向新数据（失效 SQL "能跑"）；**删源**才是引用名没了、真不可得。v1 不重算换源失效派生**不是因为不可能**，是不做失效子链重放子系统（本 ADR）。0034 Consequences 已追加指针。
- **延伸 ADR-0025**：换源失效派生在 v1 是**终局死轮**（不复活）；「接管旧源引用名」仅服务于"新轮基于新源提问"与 resume 重放，**不**服务于复活旧派生。显式重算留 v2。0025 Consequences 已追加指针。
- **闭合 ADR-0040 标的未闭合依赖**：答案——该路径在 v1 不存在。v1 不支持基于失效 SQL 的重算；用户重新提问、LLM 重新生成；失效 SQL 按既有窗口规则（窗口内可见、窗口外不发）服务，无需新机制。
- **能力边界（与 ADR-0017 同列）**：v1 不支持失效派生重算，是"v1 不做、留 v2"的 scope 边界，非"永不做"。若 v2 引入显式重算，须先解决 backing 可变语义 + 依赖链重放，届时另开 ADR 校准本决策。
- 实现侧：换源失效派生标终局 stale（不预留复活钩子）；UI 不提供"用当前源重算此派生"入口（v1）；失效 SQL 的窗口服务复用 0023 既有规则，无新代码路径。
